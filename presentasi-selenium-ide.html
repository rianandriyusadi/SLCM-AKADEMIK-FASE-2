Bagian 1 · 5 menit

Mengapa Selenium IDE?

Masalah pengujian manual

  • Tidak reproducible — dua orang menguji fitur yang sama bisa dapat hasil berbeda
  • Setiap rilis harus mengulang seluruh skenario dari nol
  • Tidak ada bukti tertulis bahwa fitur sudah diuji
  • Bug regresi baru terdeteksi setelah user komplain

Selenium IDE menjawab

  • Tidak perlu coding — gunakan antarmuka record/edit/replay
  • File .side berformat JSON, disimpan di git bersama kode
  • Siapapun dapat menjalankan test yang sama, kapanpun
  • Jika ada regresi, test langsung merah tanpa perlu cek manual

Posisi dalam workflow EUIS

Dev selesai mengerjakan fitur di branch → QA memuat file .side di Selenium IDE → menjalankan suite → jika ada test yang merah, buat screenshot → laporkan ke issue GitLab.

Demo Langsung

Langkah persiapan

  • Buka Chrome — Selenium IDE hanya berjalan normal di Chrome (lihat slide berikutnya)
  • Buka Selenium IDE (ekstensi browser)
  • Muat file: jenis-matakuliah-crud-demo.side
  • Server target: 10.119.107.223:18181
  • Pilih suite jenis-matakuliah-crud
  • Klik Run All Tests

Yang diamati bersama

  • Panel kiri: daftar test dalam suite
  • Panel tengah: daftar command per test
  • Panel bawah (Log): eksekusi real-time
  • Semua hijau = tidak ada regresi pada fitur ini
  • Ada merah = ada yang perlu ditindaklanjuti

Demo dulu — baru dijelaskan. Peserta melihat hasil akhir sebelum memahami cara kerjanya.

Bagian 3 · 15 menit

Anatomy Satu Test

Fokus: testCreateHappyPath. Dipilih karena mencakup seluruh empat blok sekaligus.

[1] LOGIN → [2] ACT → [3] ASSERT → [4] TEARDOWN

LOGIN

Logout → simpan kredensial ke variabel → login → pilih role. Setiap test berdiri sendiri — tidak bergantung state test lain.

ACT

Navigasi via menu sidebar → buka form → isi data → submit. Menguji alur navigasi sekaligus fungsi form.

ASSERT

Periksa tidak ada heading error. Periksa flash sukses tampil. Dua cek ini wajib ada di setiap test.

TEARDOWN

Hapus data test setelah selesai. Bersifat kondisional — aman dijalankan ulang meskipun test gagal di tengah.

Anatomy · Blok LOGIN

Blok [1] LOGIN

open     /site/logout
store    hening  → username
store    *****   → password   # akun test QA
open     /site/login
type     id=loginform-username  ${username}
type     id=loginform-password  ${password}
click    id=btn-site-login
waitFor  id=select-site-role
select   id=select-site-role    value=1#1
click    id=btn-site-role
waitFor  css=.navbar

Alasan desain ini

  • Logout di awal memastikan test tidak mewarisi sesi dari test sebelumnya
  • Kredensial disimpan ke variabel — jika akun test diganti, cukup ubah satu baris
  • Format role value=1#1 berarti roleId=1, institutionId=1 — format khusus EUIS
  • Tunggu css=.navbar sebelum lanjut — konfirmasi login berhasil

Blok ini identik di semua test dalam suite sehingga setiap test dapat dijalankan secara independen tanpa urutan tertentu.

Anatomy · Blok ACT

Blok [2] ACT

click    id=link-menu-konfigurasi
waitFor  id=link-conf-index-course-category
click    id=link-conf-index-course-category
waitFor  id=grid-conf-coursecategorylist

click    id=btn-conf-coursecategorylist-create
waitFor  id=form-conf-coursecategorycreate

type     id=trefjenismatakuliah-nama  0Uji188
click    id=btn-conf-coursecategorycreate

Poin penting

  • Navigasi melalui sidebar — menguji menu sekaligus fungsi halaman, bukan hanya membuka URL langsung
  • waitForElementPresent sebelum setiap interaksi — tidak ada pause dengan waktu tetap yang rawan gagal di koneksi lambat
  • Data test memakai awalan 0Uji agar selalu tampil di baris pertama grid (urutan alfabet) dan mudah diidentifikasi
  • Angka 188 adalah nomor issue GitLab yang memicu pembuatan test ini
Anatomy · Blok ASSERT

Blok [3] ASSERT

waitFor  id=grid-conf-coursecategorylist

# Pastikan tidak ada error server
assertElementNotPresent
  id=error-heading

# Pastikan flash sukses tampil
assertElementPresent
  css=.alert-success

Dua assert wajib

  • assertElementNotPresent id=error-heading — semua halaman error di EUIS merender elemen ini. Error 500 atau 403 langsung memerahkan test tanpa perlu inspeksi manual
  • assertElementPresent css=.alert-success — memverifikasi flash sukses dari server, bukan hanya redirect URL
  • Teks dalam flash tidak diuji — konten teks lebih sering berubah daripada keberadaan elemen, sehingga lebih rentan false failure
Anatomy · Blok TEARDOWN

Blok [4] TEARDOWN

storeXpathCount
  xpath=//td[text()='0Uji188']
  → cleanupCount

if  ${cleanupCount} > 0
  storeAttribute
    xpath=//tr[.//td[text()='0Uji188']]
           @data-key  → cleanupId
  click    id=btn-...-delete-${cleanupId}
  waitFor  id=modal-...-delete
  click    id=btn-modal-...-hapus
  waitFor  id=grid-...
end

Mengapa kondisional?

  • Jika test gagal sebelum data tersimpan, teardown tanpa kondisi akan error karena record tidak ditemukan
  • storeXpathCount menghitung baris yang cocok — bernilai 0 jika tidak ada
  • Blok if/end memastikan hapus hanya dijalankan jika data memang ada
  • Database selalu bersih setelah test, apapun yang terjadi selama eksekusi
Bagian 4 · Pola 1 dari 3

Dynamic ID via XPath + storeAttribute

Masalah: ID record di grid tidak diketahui saat test ditulis karena nilainya auto-increment dari database.

# Ambil ID dari atribut data-key pada baris yang cocok
storeAttribute
  target: xpath=//tr[.//td[text()='0Uji188']]@data-key
  value:  recordId

# Gunakan ID yang baru diambil untuk menargetkan tombol
click
  target: id=btn-conf-coursecategorylist-update-${recordId}

Cara membaca XPath tersebut

  • Cari semua elemen <tr> yang di dalamnya terdapat <td> berisi teks 0Uji188
  • Dari <tr> yang ditemukan, baca nilai atribut data-key
  • Simpan ke variabel recordId, lalu interpolasikan ke command berikutnya sebagai ${recordId}
Bagian 4 · Pola 2 dari 3

Conditional Teardown

Masalah: Jika test gagal di tengah jalan, kita tidak tahu apakah data sudah tersimpan di database atau belum.

# Hitung dulu berapa baris yang cocok
storeXpathCount
  target: xpath=//td[text()='0Uji188']
  value:  cleanupCount

# Hapus hanya jika data ada
if    ${cleanupCount} > 0
  ... hapus record ...
end

Kenapa ini penting

  • storeXpathCount menghasilkan 0 jika tidak ada baris yang cocok
  • Tanpa kondisi: teardown error saat data tidak ada → test dilaporkan merah meski bukan bug

Manfaat

  • Suite dapat dijalankan berulang kali tanpa error teardown
  • Database selalu bersih setelah suite selesai, apapun hasilnya
Bagian 4 · Pola 3 dari 3

Error Guard

Masalah: Error server (HTTP 500) atau akses ditolak (403) tidak selalu kentara — halaman bisa tetap tampil sebagian tanpa pesan jelas.

assertElementNotPresent | id=error-heading

Cara kerja

  • Setiap halaman error di EUIS merender <div id="error-heading">
  • Assert ini memastikan elemen tersebut tidak ada
  • Jika ada error di server, elemen muncul → assert gagal → test merah secara otomatis

Di mana dipasang

  • Setelah setiap navigasi ke halaman baru
  • Setelah submit form (sebelum assert lainnya)
  • Selalu menjadi assert pertama dalam blok ASSERT

Penutup

Ringkasan

Tanpa coding

Selenium IDE menyediakan antarmuka grafis. Menulis test berarti mengisi template yang sudah ada dan mengganti ID elemen.

Terdokumentasi

File .side disimpan di git. Siapapun dapat melihat skenario apa yang sudah diuji dan menjalankannya sendiri.

Target ke depan

Setiap merge request yang masuk ke branch main harus disertai file .side untuk fitur atau fix yang dibawa.
